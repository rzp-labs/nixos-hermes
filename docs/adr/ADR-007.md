# ADR-007: Operator-Owned CLIProxyAPI Auth State

**Status:** Accepted

## Context

The CLIProxyAPI sidecar exposes a loopback-only OpenAI-compatible gateway for Hermes inference while the host migrates away from the OMP auth-gateway path. CLIProxyAPI stores provider OAuth and refresh-token state on disk. These tokens are long-lived, operator-managed, and impractical to rotate through SOPS or recreate under `/run` on every boot.

A dedicated service user would reduce casual filesystem exposure of those token files, but it would also make provider login, token refresh repair, inspection, and manual recovery more cumbersome. The sidecar is bound to `127.0.0.1`, remote management is disabled, and local client authentication is intentionally omitted by not setting `api-keys` in the CLIProxyAPI config.

## Decision

Run `cliproxyapi-gateway.service` as `admin` and keep CLIProxyAPI auth state under `/home/admin/.cli-proxy-api`.

Treat this as an explicit operational tradeoff, not an accidental shortcut:

- provider auth state is persistent and operator-owned;
- the service remains loopback-only;
- CLIProxyAPI remote management stays disabled;
- no local `api-keys` are configured for the loopback consumer path;
- permanent host changes still flow through reviewed NixOS configuration, not imperative service edits.

## Consequences

If the `admin` account is compromised, the attacker can access CLIProxyAPI provider tokens and already has broad host authority, including the ability to request privileged NixOS operations. Moving only CLIProxyAPI to a dedicated service user would not materially address that larger compromise path.

The chosen shape keeps OAuth repair and credential inspection straightforward for the human operator. It accepts that token files live on disk in the operator home and relies on loopback binding, disabled management surfaces, reviewed declarative configuration, and host rollback as the practical controls.

A stronger future hardening path is deployment automation: after a PR is approved, merged, and published through FlakeHub, a protected automation path can trigger `nixos-rebuild switch` on the host. That would reduce the need for manual waiting while also making persistent host mutation depend on the existing PR/review/CI guardrails rather than direct interactive operator action.
